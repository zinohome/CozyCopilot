## 1. Scope

M8 ships CozyCopilot v1. The build is green, the tests are green,
the documentation exists, and someone (human) can run the app
end-to-end. Three concerns:

1. **Performance** — LCP < 2.5s on a fresh load of `/chat`. First Load JS for the home route < 150KB gzip. Static export bundle is tree-shaken and split per route.
2. **Release hygiene** — README.md, manual smoke checklist, Sentry wired but silent (no PII), CI covers all 4 build targets (web, embed, desktop-step-1, mobile-step-1).
3. **E2E** — at least 15 Playwright E2E tests covering the user-visible flows. 0 critical axe issues.

Out of scope: feature work, redesign, performance beyond LCP/JS budget, release automation (manual).

## 3. Architecture decisions

### Why Sentry DSN-gated (not always-on)

Sentry is the right tool for production observability, but it's also a third-party data sink. CozyCopilot v1 is internal — until the privacy review completes, Sentry should be **opt-in via env var**. If `SENTRY_DSN` is unset, `@sentry/nextjs`'s `init()` is skipped and the SDK is a no-op. This means:

- No accidental PII leak to a third party during development
- No console errors if DSN is malformed (Sentry is silent)
- Easy to enable in production by setting one env var

`sendDefaultPii: false` + `beforeSend` stripping are belt-and-suspenders — they catch what DSN-gating doesn't.

### Why Playwright (not just vitest-axe)

vitest-axe in jsdom catches structural a11y issues (missing labels, wrong roles) but **cannot measure real contrast, real focus, or real screen-reader behavior**. Playwright + `@axe-core/playwright` runs against the actual Next.js server-rendered output in a real browser, so it catches:

- Color contrast failures (the real ones, with real stylesheets loaded)
- Focus traps that work in jsdom but break in real browsers (race conditions on first paint)
- Layout-induced issues (offscreen elements, hidden buttons)

The 15 E2E flows also catch regressions that vitest misses — broken navigation, missing routes, server errors.

### Why `continue-on-error: true` on desktop/mobile in CI

The static-export step (`build:embed` called by `build-desktop` and `build-mobile`) works in CI. The Rust toolchain step (`cargo tauri build`) and the Capacitor sync step (Xcode/Android SDK) **do not** — they're documented in BUILD.md as platform-specific. Marking them `continue-on-error` lets the matrix stay green while we surface the platform-specific work as a follow-up.

### Why 15 E2E tests (not 50)

The plan calls for "E2E-06/15 work" — 15 tests is the threshold where coverage is real (not a toy smoke test) but the suite stays maintainable. Adding more is fast — but 15 hits the sweet spot of "covers every user-facing flow once" without becoming a maintenance burden.

## 4. Risks

| Risk | Mitigation |
|---|---|
| LCP > 2.5s after all optimizations | M8.3 records the actual measurement; if the budget is missed, document it as known and add to v1.1 backlog |
| Playwright flakes on first paint | Use `waitForLoadState("networkidle")` before assertions; `expect.poll(...)` for eventually-consistent state |
| Sentry DSN leak via env var | `SENTRY_DSN` is read in `next.config.mjs` only; never logged; never sent to client |
| E2E suite takes >10 min in CI | Split into 3 shards (auth, chat, embed); run in parallel |
| Bundle-size regression | `pnpm perf:budget` runs on every PR; non-fatal warning until M8.7 closeout |
