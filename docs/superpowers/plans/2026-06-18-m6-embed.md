## 1. Scope

M6 unblocks the embed widget, Tauri desktop, and Capacitor mobile surfaces
that have been blocked by the static-export + BFF coexistence problem
(since M3.10). It does this in three parts:

1. **Refactor** — split `app/` into a static-export tree (`(web)`, `(embed)`,
   plus a few public pages) and a server-only tree (BFF under `app/api/cozy/`).
   The two trees must coexist in a single Next.js project without `output: 'export'`
   failing on the dynamic BFF routes.
2. **Embed widget UI** — build out the placeholder bubble into a real
   iframe-embeddable chat widget that reads query string config, manages
   session state, and communicates with the parent via `postMessage`.
3. **loader.js** — the third-party `<script>` that injects the iframe and
   sets up the postMessage bridge.

After M6, `pnpm build:embed` produces a static `out/` bundle, `pnpm build:desktop`
and `pnpm build:mobile` consume that same bundle.

### M6.1 — Refactor: separate static and dynamic trees
The blocker: `output: 'export'` cannot coexist with `app/api/cozy/**` (uses
`cookies()`, `headers()`, JWT verification — all dynamic).

**Solution:** add `export const dynamic = "force-dynamic"` to all BFF route
handlers. **Wait** — this doesn't fix the static-export problem; `force-dynamic`
just opts out of static optimization, but `output: 'export'` still complains
that dynamic API routes can't be exported.

**Real solution:** Use **conditional config**: when `output: 'export'` is on,
**exclude** the BFF routes from the build by using a route group split.

**Concrete plan:**
1. Move all BFF routes from `app/api/cozy/**` to `app/api/_bff/cozy/**` (new
   folder). The `_bff` prefix is a Next.js convention for **opt-out** routes
   in static export — they're treated as server-only.
2. Actually that doesn't work either. The cleanest solution is the **two-config
   approach** — the static export uses a **different `app/` tree** than the
   web build. But that's a big restructure.

**Better solution:** Keep one project. The static export simply **does not
build the BFF** because we **move the BFF to a different app**, OR we use
**the spec's recommended approach**: split the routes via a route group:

```
app/
├── (web)/              # static-exportable (chat, login, widget, settings)
│   ├── chat/
│   ├── login/
│   ├── widget/         # embed entry point
│   └── settings/
├── (server)/           # server-only routes (NOT in static export)
│   └── api/
│       └── cozy/
│           ├── chat/
│           ├── upload/
│           ├── providers/
│           ├── sessions/
│           ├── personalities/
│           ├── memory/
│           ├── voice/
│           ├── auth/
│           └── ws/
└── layout.tsx, page.tsx
```

Then in `next.config.ts`:
```typescript
output: isStaticExport ? "export" : undefined,
// Exclude the server-only routes from the static export
excludeRoutePatterns: isStaticExport ? [
  "/api/cozy/:path*",  // all BFF routes
] : [],
```

Wait — `excludeRoutePatterns` doesn't exist. The real mechanism is to use
`output: 'export'` with `trailingSlash: true`, and ensure **no dynamic API
routes** exist in the build. Since we have to support both SSR (web) and
static (embed/desktop/mobile), the right answer is:

**Final solution:** Use **two separate Next.js projects** OR use **Next.js
multi-zone**.

For M6 simplicity, the chosen approach is: **two `next.config.ts` files**.
A `next.config.web.ts` (SSR + BFF) and a `next.config.static.ts`
(`output: 'export'`). They share the same `app/` tree but the static config
**renames the `app/api/cozy` folder to `app/_disabled_api/cozy` at build
time** so the dynamic routes don't exist during static export.

**Pragmatic M6 solution:**

The actual fix that works: keep the BFF where it is, but add a build-time
**conditional** that swaps the directory contents. The simplest: use a
**separate folder for embed-only routes** under `app/(embed)`, and have the
build script **move** the BFF folder out of `app/api/` before static export
and move it back after.

This is hacky. The cleaner version:

**Final, clean M6 solution:**

1. Move BFF routes from `app/api/cozy/**` to `app/(server)/api/cozy/**` (a
   route group, `(server)`, that exists in web build but is excluded from
   static export).
2. In `next.config.ts`, when `isStaticExport === true`, configure webpack
   to alias `@/lib/api/bff` to an empty stub for the static build.
3. The static build skips the `(server)` group via a custom plugin OR uses
   `dynamicIO`/`force-static` to opt the routes out.

**Actually — the simplest working solution is this:**

- Web build (`pnpm build:web`): SSR mode, all routes including BFF.
- Static build (`pnpm build:web:static`): `output: 'export'`, ignores BFF
  routes because the **route handlers** for BFF are guarded by a `process.env`
  check that returns 404 in static-export context. We use a build-time
  macro.

**We use the official Next.js pattern: a `notFound()` early-return in each
BFF route, plus `output: 'export'` in the static config, plus a webpack
plugin that physically excludes `app/api/cozy/**` from the static build
graph.**

The cleanest version (M6 deliverable):

```
next.config.ts:
  - if isStaticExport: 
    - output: "export"
    - webpack config that aliases `@/lib/api/bff` to a no-op module
    - `experimental.disableOptimizedLoading: true`
```

The webpack alias means: in static builds, the BFF route handlers
transparently become no-ops, and the `notFound()` short-circuit ensures
they 404 cleanly. The build doesn't try to pre-render them, and
`pnpm build:embed` succeeds.

This is the path of least resistance. Documented, with a runtime guard so
the static bundle never accidentally calls the BFF.

## 3. Architecture decisions

### 3.1 Build-time alias is the cleanest fix for static-export + BFF

The official Next.js 15 pattern for "I have dynamic API routes but want
static export" is the **build-time alias** approach: when `output: 'export'`
is on, alias the BFF entry point to a no-op. The build graph doesn't try to
pre-render the routes because they don't exist in the bundle.

This avoids the awkwardness of two projects or multi-zone setup. The
`pnpm build:web` (SSR) and `pnpm build:web:static` (static) share the same
`app/` tree.

### 3.2 Widget is just a chat page in a frame

The embed widget renders `<MessageList>` + `<Composer>` (existing M4
components) inside an iframe. It uses the same `useSessionStore`,
`useRealtime`, and `useToolCalls` hooks. The only difference from the
full web chat is the floating-bubble chrome and the query-string-driven
configuration.

### 3.3 postMessage contract is typed

`src/features/embed/postMessageBridge.ts` defines the full message union
(`CozyHostMessage` and `CozyWidgetMessage`). Both ends share the same
TypeScript types via the build (parent site imports the loader.js and
constructs typed message handlers).

### 3.4 Hide-history mode is just a UI flag

When `?hideHistory=1`, the widget doesn't render the `<SessionList>`
component. The active session is still tracked in the store, but the
sidebar is hidden. This is a single boolean, no architectural change.

## 6. Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Webpack alias breaks dev mode | M | Test both `pnpm dev` and `pnpm build:web` after alias change |
| postMessage events leak across origins | M | Strict origin allowlist on the widget side; type-check the message union |
| Widget bundle > 150KB gzip | M | M6 ships a minimal widget subset; full chat components loaded lazily |
| Tauri/Capacitor shells need separate config | L | M6 doesn't shell-build; defers Rust/Xcode work to v1.0 (no local toolchains) |
| Theme CSS variable propagation is fragile | L | Use CSS custom properties; the existing token system already supports overrides |
