## 1. Scope

M4 ships the **power-user features** that turn CozyCopilot from a basic chat app into a full CozyEngineV2 client. The 5 sub-features per the design spec are:

1. **WebSocket client** with auto-reconnect — drives real-time ToolCall visualization, typing indicators, and async-task push
2. **Async tasks** — deferred response polling + WebSocket push + OS notification on completion
3. **File upload** — drag/drop, image preview, multipart upload via the BFF `chat/voice` (actually `chat/upload`-style; the M2 BFF has `chat/voice` multipart, the upload route is M4)
4. **ToolCall viewer** — purple-strip UI, collapsible arguments/result
5. **Custom LLM provider CRUD** — base_url + api_key + model, with connection test, key encryption, and `<provider_id>:<model>` encoding in chat requests

After M4 the chat app is feature-complete from the spec's standpoint for power users. M5-M8 add voice (M5), embed widget polish (M6), themes (M7), and release polish (M8).

### M4.1 — WebSocket client (`src/lib/api/ws.ts` + tests)
- Pure TS, no React
- `class WSClient` with:
  - `connect(url, token)` — opens connection
  - `send(event)` — sends JSON
  - `on(type, handler)` — subscribe to typed events

- Auto-reconnect: exponential backoff, max 5 attempts, 1s → 2s → 4s → 8s → 16s

## 3. Architecture decisions

### 3.1 WebSocket at `src/lib/api/ws.ts` (not `src/hooks/`)
Pure TS class so it works in both Node (vitest) and browser. The React hooks (`useWebSocket`, `useAsyncTask`) wrap it.

### 3.2 `chat/async` polling vs WebSocket push — both
- Primary: WebSocket push (real-time, no polling)
- Fallback: 2s polling for `deferred` task status (when WS not connected)
- M4.2's `useAsyncTask` tries WS first, falls back to polling

### 3.3 Upload route design
- BFF `POST /api/cozy/upload` — multipart
- Forwards to CozyEngineV2 `/v1/upload` (a hypothetical route; the spec calls for "文件/图片" support — we add a new CozyEngineV2 contract OR we just have the BFF do signed-URL redirect to a separate object store; we'll fake the upstream in M4.3.5 and document the actual integration in M4.8)

### 3.4 ToolCall event format
- M2's SSE handler in `src/lib/api/chat.ts` already supports `event: error` and `event: done` — extend it to also pass through `event: tool_call` and `event: tool_result`
- These surface to React via the existing `streamChat` async generator, just with new event types

### 3.5 Custom LLM provider storage
- BFF stores the API key in the encrypted `user_provider_configs` table (CozyEngineV2 handles encryption)
- Frontend never sees the key after the initial POST — subsequent `GET` returns `{id, base_url, model, label}` without the key
- Test connection is a one-shot `POST /api/cozy/providers/test` that doesn't persist

## 6. Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| WebSocket mock in jsdom is non-trivial | M | Use a tiny in-test fake `class FakeWS implements WebSocket` (50 LOC), don't pull in `mock-socket` dep |
| CozyEngineV2 has no real `/v1/upload` route | H | M4.3.5 stubs the upstream; document the integration in M4.8 |
| ToolCall event schema in CozyEngineV2 may differ from M2's ChatStreamEvent | M | Use a type guard / runtime shape check; M2's `normalizeStreamError` pattern is the template |
| Custom provider key encryption is server-side | L | Document the contract: BFF forwards raw, CozyEngineV2 encrypts. Frontend never sees the key after POST. |
| Integration test flakiness from timing | M | Use fake timers + deterministic fixtures. 1 happy-path test only. |
