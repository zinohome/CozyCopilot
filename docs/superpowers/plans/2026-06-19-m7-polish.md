## 1. Scope

M7 turns the CozyCopilot first-version into something that feels
considered. Three concerns:

1. **5 theme presets** — Cozy Orange (default) + Calm Blue + Mint + Lavender + Mono. Both light and dark variants. Single source of truth in `src/styles/themes/*.css`, consumed by Tailwind v4 via `rgb(var(--color-X))`. Embed widget gets the same 5 themes via a shared theme registry.
2. **Theme picker** — user-facing theme + dark/light toggle in Settings, with localStorage persistence. Themes are data, not code — adding a theme = dropping a CSS file.
3. **A11y polish** — axe-core integration (vitest-axe for component tests + @axe-core/playwright for E2E), keyboard navigation audit on the chat composer / sidebar / embed widget, focus rings, aria-labels where missing.

Out of scope: per-component redesign, layout changes, copy work.
The visual rhythm stays; the colors and a11y get better.

## 3. Architecture decisions

### Why CSS-only theme switching (not React context for colors)

The current `applyTheme(theme, target)` function in M6.6 sets
inline styles on `documentElement`. That's fine for the embed
widget (single root, no SSR). For the SSR web app, we want:

- No flash of unstyled content (FOUC) when the user reloads with
  a non-default theme.
- Server-rendered HTML to use the correct theme on first paint.
- Tailwind utilities (`bg-accent`, `text-fg`) to automatically pick
  up the active theme without a `ThemeProvider` wrapper around
  every component.

The right answer is attribute-scoped CSS:

```css
:root { --color-accent: 248 123 26; }
:root[data-theme="calm-blue"] { --color-accent: 59 130 246; }
:root[data-theme="mint"]      { --color-accent: 16 185 129; }
```

Tailwind v4's `rgb(var(--color-accent))` pattern then resolves
transparently — no JavaScript color logic.

`applyTheme` (the inline-style version) stays as the embed-only
mechanism because the embed widget's `?theme=` query string can
arrive AFTER first paint.

### Why a CSS barrel + 10 small files (not 1 big file)

A single `tokens.css` with 5 themes × 2 modes = 10 sections is
hard to scan. Splitting into `themes/{name}.css` and
`themes/dark/{name}.css` means each file is ~15 lines, and adding
a theme is a 2-file change with no risk of clobbering an existing
preset's overrides.

The `@import` barrel is a one-time hit at first paint; Vite/Next
flatten it in production.

### Why `data-theme` and `data-mode` as separate attributes

A combined `data-theme="cozy-orange-dark"` would force 10
attribute values and 10 attribute selectors. Two orthogonal
attributes (`data-theme="cozy-orange"` + `data-mode="dark"`) give
10 combinations from 5+2 selectors. The CSS gets cleaner and
adding dark variants of new themes is a one-attribute change.

## 4. Risks

| Risk | Mitigation |
|---|---|
| FOUC when reloading with non-default theme | `data-theme` set via a small blocking script in `<head>` that reads localStorage before paint |
| Theme picker doesn't survive SSR | Store reads from `window` lazily; initial state matches the blocking script |
| `vitest-axe` flags a real WCAG issue we have to fix | M7.5 keyboard audit catches most of these; axe is the regression net |
| Embed widget theme override clashes with host page CSS | M6.6 already pins this — embed sets CSS variables on `documentElement`, host's own `--color-X` declarations are shadowed (specificity is the same; embed wins by last-cascade). Document the override behavior in `loader.js` comments. |
| E2E tests need a real Playwright runner | `pnpm test:e2e` is added but not blocking. The vitest-axe component tests are the primary signal. CI is M8. |
