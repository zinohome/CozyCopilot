## 1. Scope

M5 ships the **voice features** that make CozyCopilot a real-time, hands-free
chat app. Per the design spec (§6.4 stream D, §6.5 stream E):

1. **Non-realtime voice (stream D)** — push-to-talk, audio upload, BFF STT/TTS
2. **Realtime voice (stream E)** — LiveKit room, bidirectional audio, hangup
   summary upload

After M5 the chat app is feature-complete from the spec's standpoint for
**all** the AI surfaces the design called for. M6-M8 are polish, packaging, and
release.

The voice BFF route (`app/api/cozy/voice/token`) and `MicPermissionPrompt`
component (M3.4) already exist as scaffolding. M5 wires them in.

## 3. Architecture decisions

### 3.1 `livekit-client` is the unified SDK

The spec says: "livekit-client SDK 是 web/移动壳统一依赖，**不需要三套实现**"
We add `livekit-client` as a single dependency. The same `useRealtime()` hook
works in web, Tauri (WebView), and Capacitor (WebView) without platform-specific
code.

### 3.2 Frontend has zero STT/TTS code

For stream D (non-realtime), the spec says: "前端**零** STT/TTS 代码，全部走
BFF 上传音频". So `useRecorder()` only records audio, `useVoiceSend()` uploads
it, and the BFF does STT + brain + TTS, returning a `reply_audio_url` we just
play with `<audio>`.

### 3.3 Realtime state machine

Per spec §6.5: `idle → connecting → connected → active → disconnecting → disconnected`
This is implemented as a discriminated union in `useRealtime()` state. UI
components branch on state to render the right controls.

### 3.4 Realtime summary upload

Per spec: "用户挂断 → upload summary → POST /api/cozy/voice/summary {session_id, turns, tool_calls}".
LiveKit's `Room` emits `RoomEvent.Disconnected` and we can also capture track
events for `turns` and `tool_calls` aggregation. For M5, we accept a `summary`
object from the caller (UI component captures it) and POST it to the BFF.

### 3.5 Failure degradation

Per spec: "失败降级：连接失败自动降级到流 D 非实时".
When `Room.connect()` throws, the UI surfaces a "继续以文字/语音消息模式" CTA
that closes the realtime panel and lets the user use stream D.

### 3.6 Platform matrix (constraints to be aware of, not implement)

- **Web**: full support
- **Tauri 2**: needs `microphone:allow-record` in `src-tauri/capabilities/default.json` (M5 deferred to M8 verification — Rust toolchain not installed locally)
- **Capacitor iOS**: needs `NSMicrophoneUsageDescription` in `Info.plist` (already added in M3.9)
- **Capacitor Android**: needs `RECORD_AUDIO` permission (already added in M3.9)
- **iOS background lock**: known v1.0 risk per spec, document in M5 complete

## 6. Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| MediaRecorder API in jsdom is non-trivial | M | Mock via `vi.stubGlobal('MediaRecorder', ...)`, write a small fake class |
| `livekit-client` import in SSR | M | Lazy-import inside `useEffect` to avoid SSR breakage; dynamic `import('livekit-client')` |
| `livekit-client` adds significant bundle size | M | Use `next/dynamic` for `RealtimePanel` so it's only loaded when user clicks voice-call |
| Realtime hangup summary shape may differ from spec | L | M5.6 BFF route accepts `{session_id, turns, tool_calls}` as opaque JSON; documented contract |
| Browser autoplay policy blocks reply audio | M | User-gesture unlock: `useAudioPlayer` only attempts play after a user-initiated event |
| LiveKit failure cascade | M | M5.7 wraps `room.connect()` in try/catch, surfaces graceful degradation CTA |
| iOS Capacitor background lock | H (v1.0 known risk) | Document; don't implement workarounds in M5 |
